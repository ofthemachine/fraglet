# Release v0.5.0

## Key Highlights
- **`fraglet-entrypoint version`** — every binary now embeds its own provenance via `go:embed`.
- **Reproducible builds** — byte-identical binaries from the same source, Go version, and `build-info.json`.

## ✨ New Features

### Build provenance (`version` subcommand)

The entrypoint binary now carries its own lineage. Run `fraglet-entrypoint version` to see it:

```
fraglet-entrypoint v0.5.0
  commit: abc1234def56
  built:  2026-03-03T04:00:00Z
```

The `build-info.json` is generated at build time and embedded via `go:embed`. CI uses the git commit timestamp (deterministic per commit), ensuring the build-info itself is reproducible from the commit alone.

### Reproducible builds

Builds are now byte-for-byte reproducible. The `build-info.json` used for each release is published alongside the binaries. To verify a release:

```bash
git checkout v0.5.0
cp /path/to/downloaded/build-info.json entrypoint/cmd/build-info.json
cd entrypoint
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build \
  -trimpath -buildvcs=false -ldflags="-s -w" \
  -o fraglet-entrypoint-linux-amd64 ./cmd
sha256sum fraglet-entrypoint-linux-amd64   # compare against checksums.txt
```

Build flags that enable this:
- `-trimpath` — strips host-specific filesystem paths from the binary
- `-buildvcs=false` — we embed our own provenance, not Go's
- `CGO_ENABLED=0` — pure Go, no system library variance
- `-ldflags="-s -w"` — consistent symbol/DWARF stripping

The previous `-X main.commit=... -X main.buildTime=...` ldflags (which were never wired to variables in code) have been replaced by the embedded `build-info.json` approach.

## 🚀 Upgrading

Fully backward-compatible. The `version` subcommand is additive — existing `usage` and `guide` subcommands are unchanged. No config changes required.

For 100hellos containers: update the `fraglet-entrypoint` version and sha256 in the base Dockerfile (`000-base`) to point at the v0.5.0 release.
